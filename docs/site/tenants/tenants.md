# Tenants

Tenants are the entities which will use this platform. A tenant will use API resources using an API key and secret key. Tenants will have the following details stored in the application.

## Tenant Attributes

| Attribute | Type   | Requirement | Description                                      |
|-----------|--------|-------------|------------------------------------------------|
| id        | Number | Mandatory   | Identifier of the tenant on the platform.      |
| refId     | String | Optional    | Reference ID which identifies the application within the ecosystem. |
| name      | String | Mandatory   | Name of the tenant on the platform.            |
| email     | String | Optional    | Email of the tenant.                            |
| address   | String | Optional    | Address of the tenant.                          |
| city      | String | Optional    | City of the tenant.                             |
| state     | String | Optional    | State of the tenant.                            |
| country   | String | Optional    | Country of the tenant.                          |
| pincode   | String | Optional    | Pincode of the tenant.                          |

## Tenant Onboarding

Tenant onboarding refers to activities performed by the admin of the platform. As part of onboarding a tenant, the admin will perform the following actions:

1. Create a tenant.
2. Whitelist the tenant IP address.
3. Commission a tenant.

## Tenant Lifecycle

Every tenant of the platform will have a lifecycle with states related to the tenant status:

| State          | Description                                                                                                   |
|----------------|---------------------------------------------------------------------------------------------------------------|
| CREATED        | The tenant is created on the platform.                                                                       |
| CANCELLED      | Terminal state for the tenant, achievable only from the `CREATED` state.                                       |
| COMMISSIONED   | The tenant is active and can use the platform. Achieved from the `CREATED` state.                              |
| DECOMMISSIONED | Terminal state where the tenant is deactivated and no longer a user of the platform. Achieved from `COMMISSIONED`. |

## Tenant User

Once the tenant is created and commissioned, the admin of the platform can create users for the tenant with the following details:

| Attribute | Type   | Requirement | Description                                          |
|-----------|--------|-------------|------------------------------------------------------|
| id        | String | Mandatory   | Auto-generated by the system. Identifies the tenant user. |
| refId     | String | Mandatory   | Reference ID from tenant to identify the user.       |
| name      | String | Mandatory   | Name of the user.                                     |
| email     | String | Mandatory   | Email of the user.                                    |
| password  | String | Mandatory   | Password is auto-generated; can be changed by user or admin. |

## Tenant User Lifecycle

Tenant users have lifecycle states as follows:

| State         | Previous State | Description                                                                                  |
|---------------|----------------|----------------------------------------------------------------------------------------------|
| CREATED       | -              | User is created. Can be federated after this state.                                         |
| CANCELLED     | CREATED        | Terminal state for the tenant user.                                                         |
| COMMISSIONED  | CREATED        | User is ready to use application resources, authorization, and authentication features.     |
| DECOMMISSIONED| COMMISSIONED   | User is no longer authorized to use platform features on behalf of the tenant.              |

## Tenant User Authorization

Tenant user authorization generates the access token for the user. This token can be used to access resources assigned to the user through APIs and dashboard. Roles related to resources will define the features accessible.

## Tenant Authorization

Tenant authorization refers to creating an `api key` and `secret key`. Using these, the tenant application can access resources from the platform. This action is performed by a tenant user with an admin role.
